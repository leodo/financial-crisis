# 权限、安全和审计设计

状态：`Draft`

最后更新：2026-05-30

## 1. 目标

定义系统的认证、授权、secret 管理、数据访问、审计和安全基线。即使第一版是本地或内部系统，也需要从设计上避免 API key 泄露、数据篡改和配置不可追溯。

## 2. 安全边界

需要保护：

- 数据源 API key。
- 数据库连接信息。
- 原始数据和派生指标。
- 评分配置和权重。
- 预警事件操作。
- 用户账号和权限。
- 审计日志。

## 3. 角色模型

第一版建议角色：

| 角色 | 权限 |
|---|---|
| `viewer` | 查看风险面板、指标、预警 |
| `analyst` | 查看回测、添加备注、确认事件 |
| `operator` | 查看和重跑抓取任务、处理数据质量问题 |
| `admin` | 管理用户、数据源、配置和权限 |
| `service` | 服务间调用和任务执行 |

## 4. 权限矩阵

| 操作 | viewer | analyst | operator | admin |
|---|---|---|---|---|
| 查看总览 | 是 | 是 | 是 | 是 |
| 查看指标详情 | 是 | 是 | 是 | 是 |
| 查看数据源状态 | 只读 | 只读 | 是 | 是 |
| 重跑抓取任务 | 否 | 否 | 是 | 是 |
| 确认预警 | 否 | 是 | 是 | 是 |
| 修改评分配置 | 否 | 否 | 否 | 是 |
| 启停数据源 | 否 | 否 | 否 | 是 |
| 管理用户 | 否 | 否 | 否 | 是 |

## 5. 认证

第一阶段可选方案：

- 本地单用户模式：只绑定 localhost 或内网。
- 内部账号密码。
- 反向代理接入 SSO。

生产建议：

- 使用成熟身份提供方。
- API 使用短期 token。
- 服务间使用独立 service credential。

## 6. Secret 管理

要求：

- API key 不写入 Git。
- API key 不返回给前端。
- API key 不打印到日志。
- 数据源配置只保存 secret 引用。

本地开发：

- `.env`，但必须加入 `.gitignore`。

生产：

- Docker secret、Vault、云 secret manager 或等价方案。

## 7. 配置安全

敏感配置：

- 数据源启停。
- 数据源 endpoint。
- 评分权重。
- 风险阈值。
- 方法版本。
- 用户权限。

要求：

- 修改需要 admin。
- 修改进入审计日志。
- 关键配置支持导出和回滚。
- 生产变更需要 reason。

## 8. API 安全

要求：

- 管理 API 需要认证。
- 写操作需要 CSRF 或 token 防护，视部署方式决定。
- 输入参数校验。
- 分页和查询范围限制。
- 限制超大导出。
- 错误信息不暴露内部 secret 或 SQL。

## 9. 数据完整性

需要防止：

- 原始响应被覆盖。
- 已发布指标被静默修改。
- 评分结果无版本。
- 人工改动无审计。

设计要求：

- 原始响应 append-only。
- 指标修订记录 revision。
- 评分记录 method_version。
- 配置变更写 audit。

## 10. 审计日志

审计对象：

- 登录。
- 权限变更。
- 数据源启停。
- API key 轮换。
- 抓取任务手工重跑。
- 质量问题人工处理。
- 预警确认、备注、解除。
- 评分配置修改。

审计字段：

```text
audit_id
actor
actor_role
action
object_type
object_id
before_value
after_value
reason
ip_address
user_agent
created_at
```

## 11. 数据授权合规

每个数据源必须记录：

- 官方文档链接。
- 授权说明。
- 商业使用状态。
- 是否允许缓存。
- 是否允许再分发。
- 是否为原型源。

系统前端和导出功能应避免把授权受限数据公开再分发。

## 12. 备份和恢复

安全要求：

- 数据库定期备份。
- 原始数据定期备份。
- 备份加密，视环境决定。
- 定期恢复演练。
- 删除操作需要软删除或审计。

## 13. 第一版安全基线

进入实现前至少做到：

- `.env` 不入库。
- 前端不暴露 API key。
- 配置修改有审计。
- 原始响应可追溯。
- 预警事件人工操作有历史。
- 管理接口有认证。
